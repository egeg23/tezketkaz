# Чеклист пилотного запуска (Pilot Launch Checklist) — Phase 13.3.6

Сводный план запуска пилота TezKetKaz в Ташкенте. Покрывает 2 недели до
запуска и первый месяц работы. Используйте как живой документ: чекайте
галочки, добавляйте заметки, эскалируйте красные пункты.

> **Дата T-day:** объявляется руководителем проекта в Slack `#launch`.
> Все даты ниже отсчитываются от T-day.

---

## T - 2 недели: маркетинг и публикация

- [ ] Запуск Instagram / Telegram кампании (бюджет согласован финдиром)
- [ ] Публикация лендинга `tezketkaz.uz/landing` — 3 секции (магазины,
      курьеры, покупатели) с CTA на скачивание
- [ ] Установка трекинг-пикселя Meta + Yandex.Metrica
- [ ] Создание 5 коротких видео для TikTok / Reels (15 сек каждое)

- [ ] Загрузка приложений в Play Market и App Store, прохождение review

      Google Play — несколько часов до 1 дня. **Не запускать без этого.**
      План B: запустить только Android, дождаться iOS отдельно.)
- [ ] Реферальные коды раздать первым 50 бета-тестерам (-30% на 5 заказов)
- [ ] Договориться с 2 микро-инфлюенсерами (10–50k followers) — баннер
      «попробовал, удобно»
- [ ] Подготовить 1000 листовок А5 для расклейки у БЦ Inha, Westminster,
      Mega Planet, Compass Mall

## T - 1 неделя: dry-run внутренней командой

- [ ] **Все сотрудники TezKetKaz делают по 3 реальных заказа** (3 разных
      магазина, 3 разных адреса, разная оплата). Цель — обнаружить баги
      раньше клиентов.
- [ ] Внутренний канал `#dryrun-bugs` — каждая жалоба превращается в Jira
      ticket в течение часа
- [ ] Каждый магазин в пилоте получает по 5 «тестовых» заказов от нас —
      операторы магазинов учатся принимать-отдавать без боевых клиентов
- [ ] Курьеры — каждый делает 3 доставки в режиме «shadow» (см.
      `courier-onboarding.md` §6)
- [ ] **Smoke-test** (см. `smoke-tests.md`) полный прогон в проде
- [ ] **Backup-проверка** (см. `disaster-recovery.md`): пробное
      восстановление БД на staging
- [ ] Sentry, Pino-логи, Firebase Crashlytics — алерты подключены к
      Telegram-каналу `#alerts`
- [ ] Pager rotation: дежурный on-call назначен на каждые 24 часа
      первой недели; календарь в Notion `#oncall-rota`

- [ ] Резервный канал связи (на случай падения Telegram): Slack +

## T-day: день запуска

### Утренний sync (07:00, 30 мин в Zoom)

- [ ] Roll call: все ключевые роли в наличии (CEO, CTO, Ops, BD, on-call)
- [ ] Финальные no-go конфликты обсудить — если есть «красный» — отложить
      запуск на 24 часа без сожалений
- [ ] Pre-checklist прогнать по всем магазинам / курьерам
- [ ] Объявить запуск в `#launch` ровно в 09:00 UZT

### Midday check (13:00)

- [ ] Кол-во заказов за первые 4 часа vs. план (минимум 10)
- [ ] Самые частые жалобы — есть ли паттерн
- [ ] Crash-rate < 0.5%, Sentry без критичных
- [ ] Платежи проходят (Click + Payme), есть ли отказы провайдеров
- [ ] Курьеры онлайн (≥ 60% от подключённых)

### End-of-day review (22:00)

- [ ] Подсчёт: # заказов, # отменённых, средний чек, NPS первичный
- [ ] Список багов / жалоб → приоритезация в Jira
- [ ] Решение, что пушим в hotfix ночью, что — в понедельник
- [ ] Короткий пост в `#launch` для всей команды (включая ребят на
      рынке которые не в Slack)
- [ ] Поздравить команду в чате — реально

## Неделя 1: ежедневный sync

Каждый день в 10:00 — 15-минутный standup:

- [ ] Цифры вчерашнего дня (заказы, доставки, выручка, NPS)
- [ ] Открытые жалобы (≥ 1 час без ответа — эскалация)
- [ ] Технические инциденты (Sentry alerts > P2)
- [ ] Adjustments — что меняем сегодня

К концу 1-й недели:
- [ ] Прирост заказов день-к-дню ≥ 15%
- [ ] Уровень NPS ≥ 7.0 (опросом в push-уведомлении)
- [ ] Не более 1 крупного инцидента (P0 / P1)
- [ ] Каждый магазин получил ≥ 5 заказов
- [ ] Каждый курьер сделал ≥ 10 доставок

## Месяц 1: еженедельные ревью

Каждый понедельник 10:00 — 1-часовой review:

### Метрики (slide deck шаблон в Notion)

- Daily / Weekly Active Users (DAU / WAU)
- Заказы / день, средний чек, GMV
- NPS (по 5 опросам — после первого заказа, после 3-го, и т.д.)
- Жалобы по категориям (доставка, качество, оплата, приложение)
- Courier churn rate (% подключившихся, которые перестали работать)
- Shop satisfaction (опрос на ресепшене — 1-10)
- Юнит-экономика per order (выручка - расходы / заказ)

### Решения

- Включать ли новый магазин в каталог
- Выходить ли в новый район (изначально только Чиланзар + Юнусобод)
- Менять ли тарифы для курьеров
- Останавливать ли pilot и переходить на soft-launch (Phase 14)

## Эскалация во время пилота

| Уровень | Триггер                                         | Действие                                          |
|---------|--------------------------------------------------|---------------------------------------------------|
| P0      | Полная недоступность сервиса > 5 мин            | Звонок CEO + CTO. Status page обновить.           |
| P1      | Деградация (заказы не оформляются, оплата падает) | Pager on-call, фикс в течение 1 часа.            |
| P2      | Жалобы > 5 в час одного типа                    | Триаж в Jira, фикс в течение 24 часов.           |
| P3      | Косметика / редкие баги                          | Бэклог, фикс в следующем спринте.                |
